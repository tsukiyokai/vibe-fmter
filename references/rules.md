## 预处理规则（LLM语义判断）

### R0: 合并PDF提取的断行

来源：PDF文本提取实践

PDF渲染器按页面宽度在固定列截断行。pdftotext等工具提取后，语义连贯的句子被拆成多行。R0将这些"软断行"合并回段落。

主路径（SKILL.md工作流）：由LLM在运行format.py之前完成。LLM根据上下文语义判断相邻行是否属于同一段落，能正确处理"段落内句号后断行"等正则无法区分的情况。

脚本fallback（`format.py -j`）：保留基于启发式正则的合并逻辑，供直接调用脚本的场景使用。合并条件——同时满足以下全部时，将当前行与下一行拼接：
1. 当前行不是特殊行（标题、分隔符、列表项、引用、代码围栏、日期元数据）
2. 下一行非空且非特殊行
3. 当前行以"续写字符"结尾：CJK表意字符、ASCII字母/数字、或中间标点（，、；：（「『【〈《）

脚本fallback不合并的情况：
- 行以句末标点结尾（。！？⋯）→ 自然段落结束
- 行以右括号/引号结尾（）」』】〉》）→ 结构完整
- 下一行是空行 → 段落分隔
- 代码块内的行 → 保持原样

```
合并前 (PDF提取):
我最擅长的技术和最博学的知识领域始终不
是我的正路，在我业已成年

合并后:
我最擅长的技术和最博学的知识领域始终不是我的正路，在我业已成年

LLM能正确合并 (句号后断行但同一段落):
工作顺利。
单身。在这个
→ LLM判断语义上仍是同一段落，合并
→ 脚本fallback会因句号结尾而保留两行（false negative）
```

## 间距规则（脚本处理）

### R1a: CJK与非CJK内容字符之间不加空格（紧凑模式）

来源：用户偏好（覆盖"盘古之白"惯例）

CJK字符与非CJK内容字符（字母、数字、%、()、[]、~、°、@、&、_、→、←）之间不留空格，保持内容紧凑。不处理markdown语法字符(#、-、*、+、|、>)和`/`（避免破坏URL路径）以避免破坏格式。

例外：label标识符（大写字母+可选小写/连字符+数字，如C1、B-2、P0、Tier1）后跟CJK时保留空格，确保可读性。

```
正确: AI辅助HCCL通信算子组研发
错误: AI 辅助 HCCL 通信算子组研发

正确: 约30%偶然复杂性
错误: 约 30% 偶然复杂性

正确: C1 偶然复杂性占比低于20%
错误: C1偶然复杂性占比低于20%

正确: 参考[5]的数据
错误: 参考 [5] 的数据

正确: 覆盖率上限(85%)更低
错误: 覆盖率上限 (85%) 更低

正确: 不清晰→方案仅作brainstorm素材
错误: 不清晰 → 方案仅作brainstorm素材
```

### R1c: 全角标点与其他字符之间不加空格

来源：排版指北

全角标点与相邻内容字符（CJK、字母、数字、引号、半角括号、%、~、°、@、&、_、→、←）之间不留空格。不处理markdown语法字符(#、-、*、+、|、>)以避免破坏格式。

```
正确: 你好，世界
错误: 你好 ，世界

正确: 核心思想：简洁
错误: 核心思想 ：简洁

正确: 须回答：(1)并发安全
错误: 须回答： (1)并发安全
```

## 标点规则（脚本处理）

### R2a: 中文语境使用全角标点

来源：GB/T 15834

语境判断规则：标点（`,` `?` `!` `;` `:`）所属语境由其左侧紧邻字符决定。紧邻左字符为CJK→中文语境→全角。无需检查右侧（标点终结左侧子句，不属于右侧内容）。

```
正确: 什么是「中文排版」？
错误: 什么是「中文排版」?

正确: 核心结论：HCCL偶然复杂性约30%
错误: 核心结论: HCCL偶然复杂性约30%

正确: 责任方：SE
错误: 责任方: SE
```

### R2b: 英文语境使用半角标点

来源：GB/T 15834

```
正确: Use the `format` command.
错误: Use the `format` command。
```

### R2c: 不重复使用标点符号

来源：GB/T 15834

```
正确: 这太棒了！
错误: 这太棒了！！！
```

### R2d: 省略号使用六连点

来源：GB/T 15834

```
正确: 他说……
错误: 他说...
错误: 他说。。。
```

### R2e: 中文括号用全角

来源：GB/T 15834

```
正确: 版本号（v2.0）已发布
错误: 版本号(v2.0)已发布
```

### R2f: 括号配对一致

来源：GB/T 15834

开闭括号必须使用相同宽度。当不一致时，按括号内容判断：
含中文 -> 全角，纯英文/数字 -> 半角。

```
正确: God Object（9152 行）禁止整体输入
错误: God Object(9152 行）禁止整体输入

正确: evidence（TSan 输出），不能仅靠 AI
错误: evidence(TSan 输出），不能仅靠 AI

正确: (Tier 1)
错误: （Tier 1)
```

## Markdown结构规则（脚本处理）

### R5a: 标题前后保证空行

来源：CommonMark

```markdown
正确:

前一段落。

## 标题

后一段落。

错误:

前一段落。
## 标题
后一段落。
```

### R5b: 段落之间用空行分隔

来源：CommonMark

```markdown
正确:

第一段内容。

第二段内容。

错误:

第一段内容。
第二段内容。
```

### R6: 列表缩进一致

来源：CommonMark

```markdown
正确:

- 第一项
  - 子项 A
  - 子项 B
- 第二项

错误:

- 第一项
    - 子项 A
  - 子项 B
- 第二项
```

### R7: 表格列对齐

来源：GFM

```markdown
正确:

| 名称   | 说明     |
| ------ | -------- |
| foo    | 示例一   |
| barbaz | 示例二   |

错误:

| 名称 | 说明 |
| --- | --- |
| foo | 示例一 |
| barbaz | 示例二 |
```

### R8: 代码块行尾注释对齐

来源：最佳实践

同一代码块内所有行尾注释必须在同一列。对齐列 = 块内最长命令显示宽度+4空格。不同代码块各自独立对齐。仅当块内有2条及以上行尾注释时才触发对齐。

根据代码块语言标签自动选择注释标记：
- `#` 注释：bash/sh/zsh/python/ruby/yaml/toml/perl/makefile/dockerfile等
- `//` 注释：c/cpp/java/go/rust/js/ts/swift/kotlin/scala/dart/php/zig等
- 无语言标签时默认使用 `#`

```bash
正确:

cd /path/to/project    # 进入项目目录
claude                 # 启动交互式会话

错误:

cd /path/to/project    # 进入项目目录
claude                 # 启动交互式会话
```

```cpp
正确:

int x = 1;        // initialize x
int longvar = 2;  // initialize longvar

错误:

int x = 1;    // initialize x
int longvar = 2;    // initialize longvar
```

注意：
- 纯注释行（首个非空字符为注释标记）不参与对齐
- 引号内的 `#` 或 `//` 不视为注释
- 对齐列按显示宽度计算（CJK字符占2列）

### R9: 裸URL边界保护

来源：CommonMark / GFM autolink

裸URL（未用`<>`或`[]()`包裹）后紧跟全角字符（CJK或全角标点）时，自动用`<>`包裹URL，防止markdown autolink将全角字符吞入链接。

```
正确: 可运行demo：<https://godbolt.org/>（选C++17, gcc 14.2）
错误: 可运行demo：https://godbolt.org/（选C++17, gcc 14.2）
```

注意：
- 已在`<>`或`[]()`中的URL不受影响
- 代码块和行内代码中的URL不处理
- 仅当URL后紧邻全角字符（无空格）时才触发

### R10: 列表标记统一

来源：CommonMark

同文档内的无序列表标记统一为 `-`，替换 `*` 和 `+`。代码块内不处理。

```markdown
正确:

- 项目一
- 项目二
- 项目三

错误:

- 项目一
* 项目二
+ 项目三
```

## 结构层规则（Claude判断）

### S1: 标题层级不跳级

来源：最佳实践

```markdown
正确:

# 一级
## 二级
### 三级

错误:

# 一级
### 三级 (跳过了二级)
```

### S2: 同级标题不孤立

来源：最佳实践

同级标题至少出现两个，否则该层级无意义。

```markdown
正确:

## 概述
## 详情

错误:

## 概述 (整篇只有这一个二级标题)
```

### S3: 段落不超过7行

来源：庭说

超过7行的段落应拆分为多段或使用列表。

### S4: 同层级列表风格一致

来源：CommonMark

```markdown
正确:

- 项目一
- 项目二
- 项目三

错误:

- 项目一
* 项目二
+ 项目三
```
