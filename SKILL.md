# vibe-fmt

中英文混排文档格式化，三阶段管线：LLM合并PDF断行，脚本处理确定性规则，Claude检查文档结构。

## 工作流

1. 读取$ARGUMENTS指定的文件路径（支持绝对路径或相对路径）
2. 检测是否为PDF提取文本（见"PDF断行检测"）
3. 若是PDF文本→按"PDF断行合并"指令，LLM语义合并断行，写回文件
4. 执行 `scripts/format.py` 处理排版层规则R1-R10
5. 读取脚本输出的JSON摘要，了解已修改项
6. 读取脚本处理后的文件，执行结构层检查(S1-S3)
7. 如有结构问题，直接修正文件
8. 输出最终修改摘要

## 阶段0: PDF断行合并（LLM语义判断）

仅当步骤2检测为PDF文本时执行。LLM直接编辑文件，逐段合并被PDF渲染器截断的行。

指令：
- 逐段扫描，判断相邻行是否属于同一段落。属于同一段落的相邻行合并为一行
- 判断依据：上下文语义连贯性（句子未结束、语意未完成→合并）
- 保留真正的段落分隔：空行、标题、列表、引用块、代码块等结构边界
- 句号/问号/叹号结尾不一定是段落结束——如果下一行明显是同一段落的延续（同一话题、同一论点、承接词开头），仍应合并
- 不改文字内容，只改换行位置
- 代码块、表格、引用块(`>`)内的行保持原样

典型需要合并的情况：
- 行尾是CJK字符（非标点），下一行直接续写→合并
- 行尾是逗号、分号等中间标点，下一行继续→合并
- 行尾是句号，但下一行是同一段落的下一句（同话题延续）→合并
- 行尾是英文字母/数字，下一行继续同一句子→合并

典型不合并的情况：
- 两行之间有空行→段落分隔，不合并
- 下一行是标题(#)、列表(- / 1.)、引用(>)→结构边界，不合并
- 上下两行话题明显转换（不同论点、不同时间、不同事件）→分段，不合并

## 阶段1: 脚本（排版层）

运行脚本：

```bash
python3 <skill_dir>/scripts/format.py <file_path>
```

其中 `<skill_dir>` 是本SKILL.md所在目录的绝对路径。

脚本自动处理：
- R1a: 中英文/中数之间删除空格（紧凑模式）
- R1c: 全角标点与其他字符之间删除多余空格
- R2: 全半角标点按语境修正
- R2b: 括号配对修复（开闭括号全半角不一致时，按内容语种统一）
- R3: 全角数字转半角
- R4: 重复标点去重（保留……）
- R5: 标题前后空行
- R6: 列表缩进规范化（2空格/级）
- R7: 表格列对齐
- R8: 代码块内行尾 `#`/`//` 注释对齐（根据语言自动选择标记）
- R9: 裸URL后紧跟全角字符时自动用 `<>` 包裹
- R10: 列表标记统一为 `-`（替换 `*` 和 `+`）

脚本输出JSON到stdout，格式：`{"total_changes": N, "by_rule": {...}, "file": "..."}`

### PDF断行检测

PDF提取的文档（pdftotext/pymupdf等）会按页面宽度在固定列截断行。检测方法：读取文件前100行非特殊行（跳过标题、列表、空行、分隔符），统计以CJK表意字符结尾的行占比。

- 占比 > 15%: 几乎确定是PDF提取，执行阶段0
- 占比 < 5%: 正常markdown，跳过阶段0
- 5%-15%: 读几行确认是否断句不自然，酌情决定

典型症状：一个语义连贯的句子被拆成两行，第一行以汉字（非标点）结尾，第二行以汉字开头直接续写。

## 阶段2: 结构层检查

脚本完成后，读取修改后的文件，逐项检查：

- S1 标题层级不跳级：h1下不应直接出现h3，依次递进
- S2 同级标题不孤立：某级标题至少出现两次，否则该层级无意义
- S3 段落不超过7行：超长段落拆分为多段或转为列表（博客/散文类文档可跳过此项）

注：S4（列表标记统一）已移入脚本层R10自动处理。

发现问题时直接修正文件。如果某项修正需要理解语义（如段落在哪里拆分），选择最合理的拆分点。

## 输出格式

完成后输出修改摘要：

```
vibe-fmt完成:
- 排版层: N处修改 (R1间距: x, R2标点: x, ...)
- 结构层: M处修改 (S1标题层级: x, S3段落拆分: x, ...)
- 文件: <path>
```
